OpenID Connect & ID tokens

OAuth tells an app what it can do. OpenID Connect adds the missing piece: who you are. It’s the standard behind every “Sign in with…” button — a thin, well-defined layer on top of OAuth.

The one thing OAuth doesn’t do

Recall from the Why page: OAuth 2.0 is about authorization. It was never designed to answer “who is this user?” People bolted login onto OAuth anyway — by treating an access token, or a call to some profile API, as proof of identity. That was inconsistent and, in several documented cases, insecure.

OpenID Connect (OIDC) standardizes authentication properly. It is published by the OpenID Foundation as OpenID Connect Core 1.0, and it reuses OAuth’s flows wholesale — adding a few precise pieces.

// the relationship in one line

OpenID Connect = OAuth 2.0 + identity. Same redirects, same token endpoint, same Authorization Code flow — plus an ID token, a special openid scope, and a standard UserInfo endpoint.

What OIDC adds on top of OAuth

AdditionWhat it does
The openid scopeIncluding scope=openid in the request is the signal that turns a plain OAuth exchange into an OIDC one.
The ID tokenA signed JWT describing the authentication event and the user. This is the new star of the show.
The UserInfo endpointA standard API the client can call (with the access token) to fetch profile claims like name and email.
Standard claimsA defined vocabulary — sub, name, email, picture, etc. — so identity data looks the same everywhere.
Discovery & rolesA well-known metadata document, and clearer role names: the authorization server is called the OpenID Provider (OP); the client is the Relying Party (RP).

The ID token: a signed “who you are” receipt

The ID token is a JWT (JSON Web Token) — three Base64URL parts (header, payload, signature) joined by dots. It’s issued to the client, and the client verifies its signature to trust it. Decoded, the payload (the “claims”) looks like this:

id_token payload (decoded JWT)
{
  "iss": "https://accounts.example.com",   # issuer — who minted this
  "sub": "248289761001",                   # subject — STABLE user id
  "aud": "printer-app-123",                # audience — which client it's for
  "exp": 1718900000,                       # expiry
  "iat": 1718896400,                       # issued-at
  "nonce": "n-0S6_WzA2Mj",                 # replay protection
  "name": "Ada Lovelace",
  "email": "ada@example.com"
}

The most important claim is sub (subject): a stable, unique identifier for the user at this provider. Your app should key users on sub, not on email (emails change and can be reassigned).

Always validate

A client must verify the ID token before trusting it: check the signature, that iss is the expected issuer, that aud contains your client_id, that it hasn’t expired, and that the nonce matches the one you sent. Skipping these is a real-world vulnerability, not a theoretical one.

Putting it together: “Sign in with…”

An OIDC login is just the Authorization Code flow from chapter 3 with scope=openid added. The difference is what comes back:

1

App requests openid (plus a nonce)

e.g. scope=openid email profile. The nonce ties the eventual ID token to this request.

2

You authenticate with the OpenID Provider

Same login + consent screen as OAuth — you approve sharing your identity and the requested profile scopes.

3

App exchanges the code for tokens

The token response now includes an ID token alongside the access token.

4

App validates the ID token & logs you in

It checks the signature and claims, reads sub, and creates your session. Done — you’re signed in.

5

(Optional) Fetch more profile data from UserInfo

The app can call the /userinfo endpoint with the access token to get additional claims.

calling the UserInfo endpoint
GET /userinfo  HTTP/1.1
Host: accounts.example.com
Authorization: Bearer 2YotnFZFEjr1zCsicMWpAA

# → { "sub":"248289761001", "name":"Ada Lovelace", "email":"ada@example.com" }

The two-token mental model

🪪 ID token → for the app

“This user is Ada, authenticated at 14:30 by this provider.” Read by the client to establish a login session. Don’t send it to APIs.

🔑 Access token → for the API

“The bearer may read photos.” Sent to resource servers. Opaque to the client — don’t try to read identity from it.
