The Authorization Code flow (+ PKCE)

✓ Recommended Web · SPA · Mobile · Desktop RFC 6749 §4.1 · RFC 7636

This is the flow — the one you should reach for in almost every app with a user. It looks like a lot of redirects, but each step exists for a precise security reason. Let’s walk it slowly.

The one-sentence idea

The app sends you to the authorization server to log in; the server sends you back to the app with a short, single-use code; the app then quietly swaps that code (in a direct, back-channel call) for the actual tokens.

Why the two-step (code, then token) dance? So the powerful tokens never travel through the browser’s address bar where they could leak. Hold that thought — it pays off below.

Step by step

Watch it play out, then read each step below:

1

You click “Connect”

In the photo-printing app (the client), you click “Connect my photo account.”

2

The app redirects you to the authorization server

Your browser is sent to the authorization endpoint with a query string describing the request: which app, what scopes, where to return, and an anti-CSRF state.

3

You authenticate & consent

You log in directly with the authorization server (the app never sees this) and approve the requested scopes on the consent screen.

4

The server redirects back with a code

The authorization server bounces your browser back to the app’s pre-registered redirect_uri, carrying a short-lived authorization code and your state value.

5

The app exchanges the code for tokens — on the back channel

Server-to-server, away from the browser, the app POSTs the code (plus its credentials / PKCE verifier) to the token endpoint and receives an access token (and often a refresh token / ID token).

6

The app calls the API with the token

The app presents the access token to the resource server and finally fetches your photos.

What the wire actually looks like

Here’s the real HTTP, simplified. First, the redirect to the authorization endpoint (RFC 6749 §4.1.1):

1 — authorization request (browser redirect)
GET /authorize?response_type=code
            &client_id=printer-app-123
            &redirect_uri=https://printer.example/callback
            &scope=photos.read
            &state=xyz-random-anti-csrf
            &code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM
            &code_challenge_method=S256  # ← PKCE
Host: auth.photos.example

After you approve, the server redirects back (RFC 6749 §4.1.2):

2 — authorization response (redirect to the app)
HTTP/1.1 302 Found
Location: https://printer.example/callback?code=SplxlOBeZQQ&state=xyz-random-anti-csrf
# the app verifies state matches what it sent, then exchanges the code

Finally, the back-channel token exchange (RFC 6749 §4.1.3):

3 — token request (server-to-server, no browser)
POST /token  HTTP/1.1
Host: auth.photos.example
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=SplxlOBeZQQ
&redirect_uri=https://printer.example/callback
&client_id=printer-app-123
&code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk  # ← PKCE proof
4 — token response
HTTP/1.1 200 OK
Content-Type: application/json

{
  "access_token": "2YotnFZFEjr1zCsicMWpAA",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "tGzv3JOkF0XG5Qx2TlKWIA",
  "scope": "photos.read"
}

Why a code first? Why not just hand over the token?

Great question — and the reason is elegant. The authorization code comes back through the front channel (the browser redirect / URL), which is relatively exposed: URLs end up in browser history, server logs, and the Referer header. The actual tokens are far more sensitive.

So OAuth uses the code as a one-time, short-lived ticket that’s only useful when redeemed on the back channel — a direct, TLS-protected, server-to-server call where the powerful tokens are delivered out of sight. Even if someone glimpses the code in a URL, it’s single-use and useless without the follow-up exchange (and, with PKCE, without the secret verifier).

// front channel vs. back channel

Front channel = via the user’s browser (redirects, URLs) — visible, loggable.
Back channel = direct HTTPS between the app’s server and the authorization server — private.

PKCE: making this safe for phones & SPAs

The classic flow assumed a confidential client that could prove itself with a client_secret. But mobile apps and single-page apps are public clients — they can’t hide a secret. That left a gap: if an attacker intercepted the authorization code (e.g. via a malicious app registered for the same redirect), they could redeem it themselves.

PKCE (“pixie”, Proof Key for Code Exchange, RFC 7636) closes that gap with a clever one-time secret the app makes up on the spot:

A

Invent a secret (the “verifier”)

Before starting, the app generates a random code_verifier and keeps it private.

B

Send only its fingerprint (the “challenge”)

It sends code_challenge = SHA256(code_verifier) in the initial authorization request. The fingerprint is public; the verifier stays hidden.

C

Prove it at exchange time

When swapping the code for tokens, the app includes the original code_verifier. The server hashes it and checks it matches the challenge it saw earlier.

An attacker who steals the code can’t use it, because they never had the secret verifier — only its hash. Today, PKCE is recommended for all clients, public and confidential alike (RFC 9700).

Modern default

Authorization Code + PKCE is the recommended flow for web apps, SPAs, mobile, and desktop. The old Implicit flow (which returned tokens directly in the URL) is now discouraged precisely because it skipped this protection.
