OAuth 2.0 flows

A “flow” (or grant type) is the recipe an app follows to obtain a token. OAuth defines several — some are today’s best practice, others have been retired because the world found better, safer patterns. This hub covers all of them, including why the retired ones fell out of favour.

Pick the right flow in 10 seconds

The choice almost always comes down to two questions: is there a human present? and can the app keep a secret?

The recommended flows

🔑

Authorization Code + PKCE ✓ Recommended

The default for any app with a user — web, SPA, mobile, desktop. Splits a front-channel code from back-channel tokens; PKCE protects public clients.

→
⚙️

Client Credentials ✓ Recommended

No user at all. A backend service authenticates as itself to call an API — cron jobs, daemons, microservices.

→
📺

Device Authorization ✓ Recommended

For input-constrained devices — smart TVs, consoles, CLIs. You finish the login on your phone. (RFC 8628.)

→

The retired flows

These still exist in the wild and in old tutorials, so it’s worth understanding them — and exactly why modern guidance (RFC 9700) says don’t use them for new work.

🌐

Implicit grant ✗ Retired

Returned tokens directly in the URL fragment for early SPAs. No PKCE, no back channel — tokens leaked too easily. Replaced by Code + PKCE.

→
🔓

Resource Owner Password Credentials ✗ Retired

The app collected your real username & password. It reintroduced the exact “password anti-pattern” OAuth was built to kill.

→
