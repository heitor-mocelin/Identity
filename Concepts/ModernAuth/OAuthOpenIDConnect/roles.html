// 02 — Roles & vocabulary

The four roles & key terms

OAuth has its own cast of characters. Learn these four roles and a handful of words, and every diagram in the spec suddenly makes sense.

The four roles

RFC 6749 (§1.1) defines exactly four roles. Here they are with a running example: a photo-printing app that wants to print photos you store on a photo-sharing service.

Resource Owner

That’s you 🙋

“An entity capable of granting access to a protected resource.” When it’s a person, the spec calls it the end-user. You own the photos and you decide who gets to see them.

Client

The app asking for access 🖨️

“An application making protected resource requests on behalf of the resource owner.” The photo printer. Note: “client” says nothing about where it runs — server, browser, phone, or TV.

Authorization Server

The gatekeeper 🛂

“The server issuing access tokens to the client after successfully authenticating the resource owner and obtaining authorization.” It shows you the login + consent screen and hands out tokens.

Resource Server

Where your stuff lives 🗄️

“The server hosting the protected resources, capable of… responding to protected resource requests using access tokens.” The photo-sharing service’s API that actually holds your photos.

// Good to know

The authorization server and the resource server are logical roles — they’re often run by the same company (e.g. the photo service runs both), but they can be completely separate. A single authorization server can also issue tokens accepted by many resource servers.

The client never sees your password. It only ever holds a token issued by the authorization server, which it shows to the resource server to get your data.

The vocabulary you’ll keep hearing

TermPlain-language meaning
scopeThe specific permissions being requested, like photos.read. Scopes are how you grant “only this, not that.” The user sees them on the consent screen.
Authorization grantThe “permission slip” representing your approval. It’s the thing the client exchanges for a token. There are several grant types (see below).
Access tokenThe short-lived key the client uses to call the resource server. Covered in depth on the Tokens page.
Refresh tokenA longer-lived credential used to get new access tokens without bothering you again.
ConsentThe screen where you see what an app is asking for and click Allow or Deny.
redirect_uriThe address the authorization server sends you back to after you approve. Pre-registered, and tightly checked (it’s a security-critical value).
client_idA public identifier for the app, assigned when the developer registers it with the authorization server.
stateA random value the client sends and checks on return, to prevent cross-site request forgery (CSRF). More on the Security page.

Confidential vs. public clients

RFC 6749 (§2.1) splits clients into two types based on one question: can this app keep a secret?

🔒 Confidential clients

Run somewhere you control — a back-end web server. They can safely store a client_secret that proves the app’s identity to the authorization server.

🌐 Public clients

Run on the user’s device — single-page apps (SPAs), mobile apps, desktop apps. Anyone can crack them open, so they can’t hold a usable secret. These rely on PKCE instead.

Why it matters

This single distinction drives almost every “which flow should I use?” decision. A public client can’t prove who it is with a secret, so the protocol needs another way to stop attackers from stealing its codes — that’s what PKCE adds.

Grant types: the menu of flows

An authorization grant can be obtained in different ways. RFC 6749 defines four, but modern guidance (RFC 9700) has narrowed the recommended list considerably.

Grant typeUsed forStatus today
Authorization CodeWeb apps, SPAs, mobile — almost everything with a user.✓ Recommended (with PKCE)
Client CredentialsApp-to-app, no user involved (e.g. a backend job).✓ Recommended for machine-to-machine
ImplicitOld browser-based apps; returned tokens directly in the URL.✗ Discouraged — use Code + PKCE
Resource Owner Password CredentialsApp collects your username/password directly.✗ Discouraged — defeats the point

For nearly every modern app with a user, the answer is the Authorization Code flow with PKCE. That’s our next chapter.
