The Implicit flow ✗ Retired

✗ Do not use for new apps Replaced by Code + PKCE RFC 6749 §4.2 · RFC 9700

In the early 2010s, browsers couldn’t make cross-origin API calls easily and single-page apps couldn’t keep a secret. The Implicit flow was a pragmatic shortcut for that era — and it’s now retired because the shortcuts turned out to be security holes.

Don’t build new apps on this

Current best practice (RFC 9700, the OAuth 2.0 Security BCP) says the Implicit grant SHOULD NOT be used. Use the Authorization Code flow with PKCE instead — it works perfectly for SPAs today. This page exists so you understand legacy systems and why the change happened.

How it worked

The Implicit flow collapsed the two-step Authorization Code dance into one. Instead of returning a code to be exchanged on the back channel, the authorization server returned the access token directly in the redirect — in the URL fragment (the part after #). There was no token exchange, no client authentication, and no PKCE.

Notice steps 4 and 5 turn red: the moment the token lands in the URL fragment, it’s exposed.

What the wire looked like

The giveaway is response_type=token and the token coming back in the # fragment:

request & response (RFC 6749 §4.2)
# 1. authorization request
GET /authorize?response_type=token
            &client_id=spa-app-123
            &redirect_uri=https://app.example/cb
            &scope=photos.read

# 2. response — token right in the URL fragment ⚠
HTTP/1.1 302 Found
Location: https://app.example/cb#access_token=2YotnFZFEjr1zCsicMWpAA&token_type=Bearer&expires_in=3600

Why it was retired — the problems

ProblemWhy it’s dangerous
Token in the URL fragmentIt lands in browser history, can leak via the Referer header, web server logs, and analytics scripts.
Exposed to any script on the pageA single XSS flaw or malicious third-party script can read the token straight out of the page.
No client authenticationThe token is issued on the front channel with nothing proving the app’s identity.
No PKCE protectionNothing binds the response to the request that started it, enabling token injection / substitution.
No refresh tokensBy design it couldn’t issue refresh tokens, so apps used long-lived access tokens (worse if leaked) or constant re-auth.
Access token ≠ proof of identityApps that used it for “login” were exposed to token-substitution attacks — part of why OpenID Connect was created.
The root cause

Implicit optimised away the back channel — but that channel is exactly where tokens stay private and where PKCE proves the exchange is legitimate. Remove it and the powerful credential ends up in the most exposed place possible: the browser’s address bar.

What replaced it

Two things made Implicit unnecessary:

  • PKCE (RFC 7636) let public clients use the Authorization Code flow safely — no client secret required.
  • CORS matured, so SPAs can make the back-channel token request directly from JavaScript.

Together they mean an SPA can use Authorization Code + PKCE and get all of Implicit’s convenience with none of its exposure — including refresh-token support via rotation.

Migrate like this

response_type=token → response_type=code + a code_challenge. Drop the fragment-parsing; do a proper token exchange instead.
