Client Credentials flow

✓ Recommended No user · server-to-server RFC 6749 §4.4

Sometimes there’s no human in the picture — a backend job syncing data at 3am, one microservice calling another. The Client Credentials grant lets an app authenticate as itself and get a token.

The one-sentence idea

The application presents its own credentials (a client_id and client_secret) directly to the token endpoint and receives an access token scoped to the app itself — no user, no consent screen, no redirects.

Watch it happen

When to use it

✅ Great for

  • Service-to-service / microservice calls
  • Scheduled jobs, cron tasks, daemons
  • CI/CD pipelines calling an API
  • Anything acting on its own behalf, not a user’s

🚫 Not for

  • Anything representing a specific user’s data or consent
  • Public clients (SPAs, mobile) — they can’t hold a secret
  • Situations needing per-user scopes or auditing

What the wire looks like

A single back-channel request. The client authenticates with HTTP Basic (or a JSON body):

token request — RFC 6749 §4.4
POST /token  HTTP/1.1
Host: auth.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW   # base64(client_id:client_secret)
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&scope=reports.read
token response
HTTP/1.1 200 OK
Content-Type: application/json

{
  "access_token": "2YotnFZFEjr1zCsicMWpAA",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "reports.read"
}
// notice what’s missing

No redirect_uri, no authorization code, no state, and — per the spec — no refresh token (the client can just request a new one any time with its own credentials).

Security notes

  • The client_secret is the keys to the kingdom — store it in a secrets manager, never in source or a public client.
  • Scope tokens tightly: a sync job that only reads reports shouldn’t get write access.
  • Rotate secrets regularly; prefer stronger client authentication (e.g. a signed client_assertion / mTLS) over a shared secret where supported.
  • Because the token represents the app, audit logs should attribute actions to the service identity, not a user.
