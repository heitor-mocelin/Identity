Resource Owner Password Credentials ✗ Retired

✗ Do not use Reintroduces the password anti-pattern RFC 6749 §4.3 · RFC 9700

Often shortened to ROPC, this flow let an app collect your actual username and password and trade them for a token. It was meant only as a stopgap for migrating legacy apps — and it’s now retired because it undoes the very thing OAuth was invented to fix.

This defeats the purpose of OAuth

The whole point of OAuth is to stop apps from handling your password. ROPC puts the password right back in the app’s hands. RFC 9700 says it MUST NOT be used. Prefer Authorization Code + PKCE.

How it worked

The app showed its own login form, you typed your real credentials into it, and the app POSTed them to the token endpoint with grant_type=password. In return it got tokens — but only by seeing, handling, and potentially storing your password along the way.

Steps 1, 2 and 4 are red: your password is exposed to the app, and the protections of a real login (consent, MFA, federation) are all bypassed.

What the wire looked like

token request (RFC 6749 §4.3)
POST /token  HTTP/1.1
Host: auth.example.com
Content-Type: application/x-www-form-urlencoded

grant_type=password
&username=ada@example.com      # ⚠ the user's real login
&password=hunter2-for-real      # ⚠ the user's real password, seen by the app
&scope=photos.read

Why it was retired — the problems

ProblemWhy it’s dangerous
The app sees your passwordIt can log it, store it, reuse it, or leak it in a breach — the exact “password anti-pattern” OAuth set out to eliminate.
No consent screenThere’s no moment where you see and approve what the app may access; it’s all-or-nothing.
Breaks MFAA simple username/password POST can’t carry a second factor, push approval, or step-up challenge.
Breaks federation / SSOIf you log in via an external identity provider or social login, there’s no password for the app to collect at all.
Trains bad habitsIt conditions users to type their primary password into any app — a phishing dream.
Wide blast radiusOne compromised app means a compromised password, and every service that shares it.

What to use instead

For interactive logins, use Authorization Code + PKCE: the user authenticates directly with the authorization server (which can enforce MFA, SSO, and consent), and the app only ever receives tokens — never the password.

❌ ROPC

  • App handles your password
  • No consent, no MFA, no SSO
  • Phishing-friendly habit

✅ Authorization Code + PKCE

  • You log in at the auth server
  • MFA, SSO & consent all work
  • App sees only scoped tokens
// edge case

The only places ROPC was ever tolerated were tightly-controlled first-party migrations off legacy systems. Even there, modern guidance is to move to a redirect-based flow as soon as possible.
