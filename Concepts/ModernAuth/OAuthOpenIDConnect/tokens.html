Tokens: access, refresh & bearer

Tokens are the currency of OAuth. Here’s what they actually are, the different kinds, how they travel across the wire, and why their short lifetimes are a feature, not a bug.

What is a token, really?

An access token is, in the words of RFC 6749, simply “a string representing an access authorization issued to the client.” That’s it — a string. It encodes (directly or by reference) a specific scope, lifetime, and other access attributes.

Think of it as a concert wristband. The box office (authorization server) checked your ID and payment once; now the wristband alone gets you through every gate. Security staff (the resource server) just glance at the band — they don’t re-run your whole purchase.

// the abstraction (RFC 6750 §1.3)

“The access token provides an abstraction, replacing different authorization constructs (e.g., username and password) … This abstraction enables issuing access tokens valid for a short time period, as well as removing the resource server’s need to understand a wide range of authentication schemes.”

Two shapes of access token

📦 Self-contained (e.g. JWT)

The token carries its own data — scopes, expiry, subject — signed so the resource server can validate it locally without a network call. Fast, but harder to revoke instantly.

🎫 Reference (“opaque”)

The token is just a random handle. The resource server (or a token-introspection endpoint) looks it up to find what it grants. Easy to revoke, but needs a lookup.

OAuth 2.0 itself doesn’t mandate a format — both are valid. The resource server and authorization server just need to agree.

When the token is a JWT, it’s three Base64URL parts joined by dots — header, payload, signature:

“Bearer” tokens: whoever holds it, wields it

Most access tokens are bearer tokens, defined in RFC 6750. The name is the warning label:

Bearer = cash

“Any party in possession of a bearer token (a ‘bearer’) can use it … without demonstrating possession of a cryptographic key.” Like a $20 bill: whoever holds it can spend it. So bearer tokens must be protected in storage and in transit — which is why TLS is mandatory.

RFC 6750 specifies how to send a bearer token. The strongly preferred way is the HTTP Authorization header:

calling the resource server — RFC 6750 §2.1
GET /photos  HTTP/1.1
Host: api.photos.example
Authorization: Bearer mF_9.B5f-4.1JqM

RFC 6750 also allows a form-encoded body parameter or a URL query parameter — but the query-string method is discouraged, because tokens in URLs leak into logs, history, and referrers.

If the token is missing or bad, the resource server replies with a WWW-Authenticate header and one of three error codes:

ErrorMeaningHTTP status
invalid_requestThe request itself is malformed.400
invalid_tokenThe token is expired, revoked, or bogus.401
insufficient_scopeValid token, but it lacks the scope for this action.403

Why access tokens expire so fast

Access tokens are deliberately short-lived — often minutes to an hour. That seems inconvenient, but it’s a core security feature: if a bearer token does leak, the damage window is small. A stolen token that expires in 15 minutes is far less valuable than a password that works forever.

But you don’t want to be redirected to log in every 15 minutes. That’s where the second token comes in.

Refresh tokens: staying signed in, safely

A refresh token is a longer-lived credential whose only job is to obtain new access tokens when the old one expires — quietly, on the back channel, without involving you.

refreshing an access token — RFC 6749 §6
POST /token  HTTP/1.1
Host: auth.photos.example
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=tGzv3JOkF0XG5Qx2TlKWIA
&scope=photos.read   # may request equal-or-narrower scope

Because refresh tokens live longer, they’re higher-value targets and get extra protection. RFC 6819 and modern practice recommend:

  • Bind to the client — a refresh token only works for the client_id it was issued to.
  • Rotation — each time you refresh, issue a new refresh token and invalidate the old one. If an old one is reused, that signals theft and the whole chain can be revoked.
  • Revocability — let users and servers kill a refresh token (e.g. “sign out everywhere”).
  • Restricted issuance — public clients should only get refresh tokens under stricter conditions.
Mental model

Access token = a day pass you flash repeatedly but that expires tonight. Refresh token = your membership card, kept safe, used occasionally to get a fresh day pass.

Don’t confuse access tokens with ID tokens

One more token is coming in the next chapter: the ID token from OpenID Connect. Keep them straight:

TokenAnswersAudience (who reads it)
Access token“What may this app do?”The resource server / API
Refresh token“Get me a new access token.”The authorization server
ID token“Who is this user, and how did they log in?”The client app itself
Don’t do this

An access token is not proof of who the user is. It’s opaque to the client and meant for the API. Using it as a login assertion is a classic mistake — that’s the ID token’s job.
