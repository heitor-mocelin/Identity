Device Authorization flow

✓ Recommended TVs · consoles · CLIs · IoT RFC 8628

Ever signed into a streaming app on a TV by visiting a URL on your phone and typing a short code? That’s this flow. It exists for devices that can’t show a browser or take typed passwords easily.

The one-sentence idea

The device gets a short user code and a verification URL from the authorization server, shows them on screen, and polls in the background while you complete the login on a second device (your phone or laptop). When you approve, the device’s next poll returns a token.

Watch it happen

Why a second device?

Typing a password with a TV remote is miserable, and many devices have no browser to run a normal redirect-based login. The Device Authorization grant cleverly moves the hard part — authentication and consent — onto a device you already trust and that has a real keyboard. The constrained device only ever displays a code and waits.

What the wire looks like

Step 1 — the device asks to begin (RFC 8628 §3.1–3.2):

1 — device authorization request
POST /device_authorization  HTTP/1.1
Host: auth.example.com

client_id=tv-app-9000&scope=profile

# response:
{
  "device_code": "GmRhmhcxhwAzkoEqiMEg_DnyEysNkuNhszIySk9eS",
  "user_code": "WDJB-MJHT",
  "verification_uri": "https://example.com/device",
  "expires_in": 1800,
  "interval": 5
}

The device shows: “Go to example.com/device and enter WDJB-MJHT.” Meanwhile it polls (step 5), respecting interval seconds between tries:

5 — device polls the token endpoint
POST /token  HTTP/1.1

grant_type=urn:ietf:params:oauth:grant-type:device_code
&device_code=GmRhmhcxhwAzkoEqiMEg_DnyEysNkuNhszIySk9eS
&client_id=tv-app-9000

# while you haven't approved yet:
{ "error": "authorization_pending" }     # keep polling
# after you approve:
{ "access_token": "2YotnFZFEjr1zCsicMWpAA", "token_type": "Bearer", "expires_in": 3600 }
// polite polling

The server can answer authorization_pending (not done yet) or slow_down (poll less often). The device must honour the interval so it doesn’t hammer the endpoint.

Security notes

  • Short, human-friendly user codes with limited lifetime and rate-limiting, so they can’t be guessed or brute-forced.
  • The verification page should clearly show which device and app you’re authorizing, to resist social-engineering (“enter this code” scams).
  • The device_code is the secret that actually redeems the token — it stays in the back channel, never shown to the user.
  • Bind the issued token to least-privilege scopes; many devices only need read access.
