Threats & best practices

Every odd-looking detail in OAuth — the state parameter, the code/token split, strict redirect matching — exists to stop a specific attack. This chapter shows the attacker’s view from RFC 6819 and the defenses that answer it.

The security mindset

RFC 6819 — OAuth 2.0 Threat Model and Security Considerations — assumes a hostile world: attackers can read network traffic that isn’t encrypted, register their own malicious clients, trick users into clicking links, and run scripts in browsers. The protocol’s job is to stay safe anyway. Three principles run through all of it:

⏳ Least lifetime

Tokens and codes expire fast, shrinking the damage window if one leaks.

🎯 Least privilege

Scopes and audiences keep each token narrow — it can do exactly one job, nowhere else.

🔗 Tight binding

Codes and tokens are bound to a client, a redirect URI, and a session — so a stolen one doesn’t fit elsewhere.

The headline threats & their defenses

1 · CSRF on the redirect (session fixation)

Attack: an attacker starts an authorization flow, captures their own code, and tricks your browser into completing it — silently linking your session to the attacker’s account (or vice-versa).

Defense — the state parameter

The client generates a random, unguessable state, sends it in the request, and verifies the exact same value comes back. A forged callback won’t carry the right state and is rejected. In OIDC, the nonce plays a similar role for the ID token.

2 · Authorization code interception

Attack: on mobile, a malicious app registers the same custom redirect scheme and grabs the code as it comes back, then redeems it for tokens.

Defense — PKCE

The code is useless without the secret code_verifier that only the legitimate app holds (see the Authorization Code flow page). Stealing the code alone gets the attacker nothing.

3 · Open redirectors & redirect-URI manipulation

Attack: if the authorization server is sloppy about where it sends users back, an attacker crafts a request that redirects the code to a site they control.

Defense — exact, pre-registered redirect URIs

Servers must register redirect URIs in advance and match them exactly (no wildcards, no partial matches). The code is also bound to the redirect URI used to request it.

4 · Token leakage in transit & in URLs

Attack: bearer tokens travel over an unencrypted link, or sit in a URL that ends up in browser history, server logs, or the Referer header — and get scooped up.

Defense — TLS everywhere + header, not URL

TLS is mandatory for every OAuth exchange. Send tokens in the Authorization: Bearer header, never the query string. And because they’re bearer tokens, keep them out of logs entirely.

5 · Phishing via counterfeit servers

Attack: a fake authorization-server page harvests the user’s real password; or an app embeds a web-view it can read.

Defense — server authentication & system browsers

Users (and apps) must verify they’re talking to the genuine server via TLS certificates. Native apps should use the system browser, not an embedded web-view the app can spy on.

6 · Replay & theft of long-lived refresh tokens

Attack: a refresh token is stolen and reused to mint endless access tokens.

Defense — rotation, binding, revocation

Rotate refresh tokens on every use and detect reuse of an old one (a theft signal); bind each to its client_id; and support revocation so a compromised token can be killed.

What modern guidance says (RFC 9700)

The IETF later distilled years of experience into RFC 9700, Best Current Practice for OAuth 2.0 Security. Its headline recommendations:

  • Use Authorization Code + PKCE for all clients — public and confidential.
  • Do not use the Implicit grant (tokens in the URL fragment) anymore.
  • Do not use the Resource Owner Password Credentials grant — it reintroduces password-sharing.
  • Exact redirect-URI matching, every time.
  • Sender-constrain tokens where possible (e.g. DPoP / mTLS) so a stolen bearer token can’t be replayed.
  • Rotate refresh tokens and detect reuse.
If you remember one thing

The two “convenient” legacy flows — Implicit and Password — are now actively discouraged. Reach for Authorization Code + PKCE, keep tokens short-lived and out of URLs, and validate everything that comes back.

You’ve reached the end 🎉

You now have the full mental model:

  • Why — to stop password-sharing, via scoped, expiring, revocable tokens.
  • Who — four roles: resource owner, client, authorization server, resource server.
  • How — the Authorization Code flow with PKCE, splitting code (front channel) from token (back channel).
  • What — access, refresh, and bearer tokens; ID tokens for identity via OpenID Connect.
  • Safely — defenses mapped to a concrete threat model.
