Security & best practices

SAML’s safety rests entirely on signatures verified correctly and assertions used exactly once, in the right place, at the right time. Here’s the attacker’s view and the checks that defeat them.

The security mindset

The OASIS Security and Privacy Considerations document assumes attackers can observe and tamper with browser traffic, lure users to malicious pages, and replay captured messages. Because SAML assertions are usually bearer tokens (whoever holds one is treated as the user), the whole model leans on three pillars:

🔏 Verify signatures

Trust only assertions signed by the expected IdP — and verify the exact element you rely on.

⏳ Tight time & use

Short validity windows + one-time use shrink the replay window to near zero.

🎯 Bind to context

Audience, recipient and request-correlation ensure an assertion only works here, for this request.

The headline threats & their defenses

1 · XML Signature Wrapping (XSW)

Attack: the classic SAML attack. An attacker takes a legitimately signed assertion and restructures the XML so that the signature still validates against the original element, while the SP’s code actually reads a different, attacker-injected assertion elsewhere in the document.

Defense — verify what you consume

Validate the signature and then process only the exact element that was signed (by reference/ID), using a hardened, well-tested SAML library. Reject documents with multiple or misplaced assertions. Never trust an unsigned assertion.

2 · Assertion replay

Attack: an attacker captures a valid assertion (e.g. from logs or a leaky referrer) and submits it again to impersonate the user.

Defense — short windows, one-time use, InResponseTo

Enforce NotBefore/NotOnOrAfter (a few minutes), remember assertion IDs to reject reuse, and for SP-initiated flows require InResponseTo to match a request you actually sent.

3 · Man-in-the-middle / eavesdropping

Attack: messages travel over plain HTTP and get read or altered in transit.

Defense — TLS everywhere

All SAML browser exchanges must run over HTTPS/TLS, end to end. This protects messages in transit and authenticates the endpoints.

4 · Audience / recipient confusion (token redirect)

Attack: an assertion minted for App A is replayed against App B.

Defense — check Audience & Recipient

The SP must reject any assertion whose AudienceRestriction isn’t itself and whose Recipient isn’t its own ACS URL.

5 · Unsolicited (IdP-initiated) response abuse

Attack: because IdP-initiated responses aren’t tied to an SP request, they’re easier to forge, replay, or use in login-CSRF.

Defense — prefer SP-initiated; harden ACS

Favor SP-initiated SSO so responses carry InResponseTo. If you must accept IdP-initiated responses, apply strict replay, timing and audience checks, and protect against login CSRF.

6 · Open redirect via RelayState

Attack: RelayState is attacker-controlled and used to bounce the user to a malicious site after login.

Defense — validate RelayState

Treat RelayState as untrusted input: allow only known internal paths, never arbitrary external URLs.

Practical hardening checklist

  • Use a mature, maintained SAML library — don’t hand-roll signature validation.
  • Require signatures and reject unsigned assertions/responses.
  • Enforce short validity windows and a replay cache of seen assertion IDs.
  • Validate Issuer, Audience, Recipient, timing, and InResponseTo every time.
  • TLS everywhere; keep assertions out of URLs, logs, and referrers.
  • Rotate certificates on a schedule and watch metadata expiry.
  • Prefer SP-initiated SSO; allow-list RelayState.

SAML vs OAuth 2.0 / OpenID Connect

They overlap but were built for different worlds. Use this to choose — and to translate between them:

 SAML 2.0OAuth 2.0 + OpenID Connect
Primary jobAuthentication + SSO (and attributes)OAuth: authorization · OIDC: authentication
FormatXML assertionsJSON / JWT tokens
TransportBrowser redirects & form POSTHTTPS redirects + REST/JSON APIs
Trust mechanismXML-DSig + exchanged metadata/certsSigned JWTs + discovery/JWKS
Best fitEnterprise web SSOWeb, mobile & API ecosystems
Role namesIdP · SP · principalOP/Auth server · client/RP · resource owner
“Token” equivalentAssertionID token (identity) · access token (API)

You’ve reached the end 🎉

The full SAML mental model:

  • Why — to end password sprawl with federated single sign-on.
  • Who — principal, Identity Provider, Service Provider, joined by pre-established trust.
  • How — Web Browser SSO: AuthnRequest via Redirect, signed Response via POST.
  • What — the signed XML assertion: subject, conditions, authn & attribute statements.
  • Safely — verify signatures correctly; bind by audience, recipient, time and request.
