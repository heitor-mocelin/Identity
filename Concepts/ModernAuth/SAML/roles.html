Roles & key terms

SAML has a small cast and a handful of special words. Learn these and the XML diagrams in the OASIS spec suddenly read like a story.

The three players

We’ll keep a running example: you, an employee, opening a SaaS expense tool that your company connects via SAML.

Principal

That’s you 🙋

The subject being authenticated — the human (or system) trying to access a service. Also called the subject.

Identity Provider (IdP)

The front desk 🛂

Authenticates the principal and issues signed assertions about them. Run by your company / its identity platform.

Service Provider (SP)

The app 🗄️

The application you want to use (the expense tool). It relies on the IdP’s assertion to log you in — so it’s also a “relying party.”

// mapping to OAuth/OIDC

If you’ve read the OAuth guide: the SAML IdP is like the OpenID Provider / authorization server, and the SAML SP is like the relying party / client. Different words, similar jobs.

The crucial word is trust. Before any user ever logs in, the IdP and SP exchange configuration so each knows and trusts the other. That setup is captured in metadata.

The vocabulary you’ll keep hearing

TermPlain-language meaning
AssertionThe signed XML “key card” the IdP issues about you — the heart of SAML. Detailed in the Assertions chapter.
AuthnRequestThe message an SP sends to the IdP meaning “please authenticate this user.”
ResponseThe IdP’s reply to the SP, which contains the assertion.
MetadataAn XML document describing an IdP or SP: its URLs (endpoints), its signing certificate, and supported options. Exchanged up front to establish trust.
BindingHow a SAML message is transported over HTTP — e.g. HTTP-Redirect (in the URL) or HTTP-POST (in a form). See the SSO Flow chapter.
ProfileA recipe combining messages + bindings for a use case. The big one is Web Browser SSO.
NameIDThe identifier for the subject inside an assertion (e.g. an email or an opaque, persistent ID).
ACS (Assertion Consumer Service)The SP endpoint that receives and processes the assertion.
AttributesFacts about the user carried in the assertion — name, email, group/role — used by the SP for authorization.
RelayStateAn opaque value used to remember where the user was headed, so they land on the right page after SSO.

Trust: metadata & certificates

SAML trust isn’t magic — it’s set up deliberately. When connecting an SP to an IdP, administrators exchange metadata, which includes the public X.509 certificate used to verify signatures.

idp-metadata.xml (trimmed)
<EntityDescriptor entityID="https://idp.example.com/saml">
  <IDPSSODescriptor ...>
    <KeyDescriptor use="signing">
      <!-- the IdP's public certificate; SPs use it to verify signatures -->
      <X509Certificate>MIIC8jCCAdqgAwIBAgI...</X509Certificate>
    </KeyDescriptor>
    <SingleSignOnService
        Binding="...HTTP-Redirect"
        Location="https://idp.example.com/sso"/>
  </IDPSSODescriptor>
</EntityDescriptor>

Because the SP holds the IdP’s certificate in advance, it can cryptographically verify that an incoming assertion really came from the IdP and wasn’t tampered with.

Bindings & profiles, briefly

Bindings = the “how”

HTTP-Redirect: message compressed into a URL query (good for short requests).
HTTP-POST: message sent as a hidden form auto-submitted by the browser (used for the bigger, signed assertion).
Artifact: send a small reference, then fetch the real message over a back channel.

Profiles = the “recipe”

The Web Browser SSO Profile is the one you’ll meet 95% of the time: an SP sends an AuthnRequest via Redirect; the IdP returns a Response via POST. Also defined: Single Logout (SLO) to end sessions everywhere.
